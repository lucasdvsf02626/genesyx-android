### A6 — Defect found (real, not stylistic)

**The reminder toggle lies to new users.**

`pushEnabled` defaults to **`true`** — `GenesyxPreferencesDataStore.kt:46` (`it[Keys.PUSH] ?: true`)
and `PreferencesRepository.kt:30` (`Eagerly, true`). But work is only ever enqueued inside
`ProfileViewModel.setPush()`, which only runs when the user *touches* the switch.

So on a fresh install: Profile renders **"Daily check-in reminder: ON"**, no `POST_NOTIFICATIONS`
has ever been granted, no work has ever been enqueued, and **no reminder will ever arrive.** The
switch says on; the feature is off.

This directly contradicts the code's own doc comment at `ProfileScreen.kt:548` — *"the switch can
never sit 'on' while the system silently drops every notification"* — which is precisely what
happens on first run. It also breaks invariant 4 ("nothing fires before the user has enabled").

Fix is one character (`?: false`), but it belongs to whoever owns this branch. Flagging, not fixing.

### Part B — Gap matrix

Everything marked EXISTS in the task brief was verified against the code. Two of the brief's claims
about the existing reminder are **wrong**:

- Brief: *"user-set time (default 19:00)"* → **Reality: hardcoded 20:00, no picker anywhere.**
- Brief: *"two-branch: nothing meaningful logged → log invitation; else water < goal → hydration

**Priority order for same-day collision** — I agree with the brief's proposal and would ship it as:
`daily log reminder > streak milestone > cycle heads-up > weekly digest > pH nudge`, sleep never
competing (it lives inside the digest). One caveat: a 30-day streak milestone losing to a routine
daily nudge feels wrong. Suggest milestones outrank the daily reminder **on the milestone day only** —

tomorrow at no cost. Human to rule.

**Android has no banned-phrase scan** to port — the brief was right to hedge. It is net-new, and it
is cheap: a pure function over a string list, unit-tested, called by the gate. Worth doing.
